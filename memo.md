# Forwarding Hub - OCR 및 데이터 병합 트러블슈팅 메모

## 1. 문제 상황
화주가 차량 말소증 사진과 차대번호 각인 사진을 동시에 여러 장 업로드했을 때, 차량대장(데이터 표)에 차대번호(VIN)만 인식되어 들어가고 **나머지 정보(등록번호, 차종, 주행거리, 연식 등)가 전부 빈 칸**으로 나오는 현상 발생.

## 2. 원인 분석 및 해결 과정

### Step 1: 백엔드 DB 업데이트 누락 버그 (수정 완료)
- **원인:** 백엔드의 `fileController.ts`에서 두 번째 사진(말소증)을 분석할 때, DB에 이미 같은 차대번호가 있으면 기존 데이터를 수정(UPDATE)하게 되는데, 이 때 코드가 `deregistration_no`(등록번호) 하나만 업데이트하고 나머지 추출된 주행거리나 연식 등은 DB에 저장하지 않고 버리고 있었음.
- **해결:** 말소증(문서) 타입으로 인식되면 `vehicle_type`, `mileage`, `initial_registration_date`, `make`, `model`, `year` 등의 필드도 모두 DB에 업데이트(UPDATE)하도록 로직 추가.

### Step 2: 5000번 포트 충돌 (EADDRINUSE) (해결 완료)
- **원인:** 백엔드 코드를 수정한 후 `nodemon`이 서버를 자동 재시작하는 과정에서, 기존 프로세스가 완전히 죽지 않은 상태로 새 프로세스가 포트 5000을 점유하려 하여 충돌 발생.
- **해결:** 백그라운드 프로세스 확인 후 `kill -9`를 통해 좀비 프로세스를 날리고 백엔드 서버를 깔끔하게 재구동.

### Step 3: OCR 정규식(문자 인식 패턴) 인식률 문제 (개선 완료)
- **원인:** 
  1. 말소증 사진 원본에 적힌 등록번호가 '90머 9212'였으나, 사진이 흐리거나 인쇄가 얇아 구글 비전 API(AI)가 한글을 날려버리고 `909212` 숫자로만 텍스트를 반환함. 
  2. 기존 정규식은 한글이 포함된 번호판만 찾도록 엄격하게 짜여 있어 데이터를 놓침.
  3. 주행거리 역시 `249902` 숫자만 있고 `km` 표기가 없어 놓침.
- **해결:** 
  - `ocrService.ts`의 정규식을 수정하여, 주변 문맥(예: '등록번호', '차종 및 주행거리' 등)을 기반으로 한글이 누락된 숫자형 등록번호와 `km` 표기가 빠진 주행거리도 유연하게 잡아내도록 개선.
  - 추가로, 흑백 필터(Grayscale) 전처리 과정이 오히려 흐릿한 글씨를 지워버리는 악영향이 있어 필터를 제거함.

### Step 4: 프론트엔드 React State 병합(Merge) 누락 (최종 해결 완료)
- **원인:** 백엔드와 인공지능을 완벽히 고쳤음에도 화면에서 여전히 빈 칸이 나타남. 진짜 원인은 프론트엔드의 `VehicleDashboardModal.tsx` 파일에 있었음.
- **분석:** 여러 장의 사진 결과를 `forEach`로 돌면서 상태(`vehicles`)를 업데이트할 때, **"이미 추출된 차대번호(VIN)가 리스트에 존재하면, 새로 들어온 사진의 데이터는 그냥 버린다"**라는 로직이 짜여 있었음. 이로 인해 1번 사진(차대번호 사진)에서 차량 데이터가 생성되면, 2번 사진(말소증 사진)에서 추출된 알짜배기 정보들이 전부 버려지고 있었음.
- **해결:** `newVehiclesMap`과 `setVehicles` 로직에서 동일한 차대번호가 발견될 경우, 기존 객체를 무시하지 않고 새로 추출된 제원 정보(등록번호, 차종, 주행거리 등)를 **기존 데이터 위에 덮어쓰기(Merge)** 하도록 코드를 전면 수정함.

## 3. 결론
결과적으로 백엔드의 DB 저장 로직, AI 문자열 추출(정규식) 패턴, 프론트엔드의 상태 병합 로직 3군데 모두에 존재하던 헛점들을 완벽하게 보완 및 수정하였음. 이제 화주가 여러 장의 사진을 순서 상관없이 올리더라도, 추출된 모든 정보가 유실 없이 병합되어 대시보드 표에 정상적으로 나타남.
