# 물류 FMS 아키텍처 설계: 선적 관리와 차량 탁송 UI 분리 전략

## 1. 개요 (Overview)
로로선(Ro-Ro)을 이용한 차량 수출 특성상, 1개의 선하증권(B/L)에 수십 대의 차량(VIN)이 묶이고 각 차량의 탁송 방식(카캐리어, 인탁송, 셀프로더)과 내륙 발생 비용이 제각각 분리되는 **극단적인 1:N 데이터 구조**를 가집니다. 

이에 따라 '해상 운임/스케줄'을 관리하는 탭과 '내륙 차량 배차/정산'을 관리하는 탭을 시스템적으로 분리하는 아키텍처를 채택합니다.

---

## 2. UI/UX 분리 구조 비교 및 결정 사유

### 🚨 기존 안: 선적 관리 탭 통합형
* **문제점:**
  * **렌더링 과부하:** 선적 기본 정보와 N대의 차량 배차 정보, 개별 할증 비용이 한 화면(DOM)에 렌더링되어 성능 저하 발생.
  * **상태 관리 복잡도:** B/L 상태 업데이트 시 차량 배열 깊은 곳의 데이터까지 영향을 받아 상태 관리 난이도 급증.
  * **UI 피로도:** 포워더의 해상 스케줄 담당자와 내륙 배차 담당자의 작업 영역이 혼재됨.

### ✅ 확정 안: 차량 운송(배차) 전용 탭 분리형
* **기대 효과:**
  * **Data Grid 최적화:** 차량 배차와 내륙 정산에만 집중할 수 있는 대형 테이블 UI 구성 가능. (체크박스를 통한 카캐리어 일괄 배차 등)
  * **트랜잭션 분리:** 해상 운임(B/L 마스터) 업데이트와 내륙 비용(Surcharge) 업데이트 API를 명확히 분리하여 트랜잭션 충돌 방지.
  * **작업 동선 분리:** 담당자 역할에 맞는 독립적인 작업 공간 제공.

## 3. 디렉토리 구조 (Directory Structure)
기존 프로젝트 구조에 맞춰 프론트엔드(Next.js App Router)와 백엔드(Node.js)의 디렉토리를 물리적, 논리적으로 분리하여 구성합니다. `api` 폴더는 `src` 외부에 위치합니다.

```text
project-root/
├── api/                        # Node.js Express 백엔드 (src 외부)
│   ├── routes/
│   │   ├── shipmentRoutes.js   # 선적/해상 마진 관련 라우터
│   │   └── dispatchRoutes.js   # 내륙 탁송/배차 관련 라우터 (분리)
│   └── controllers/
├── src/                        # Next.js 프론트엔드
│   ├── app/
│   │   ├── shipments/          # 선적 관리 탭 (해상 스케줄, 포워딩 마진)
│   │   └── dispatches/         # 차량 운송 관리 탭 (차량 리스트, 배차)
│   ├── components/
│   │   └── DataGrid/           # 차량 목록 일괄 처리를 위한 공통 그리드
│   └── store/                  
│       └── useFmsStore.js      # Zustand 상태 관리
└── .env                        # DB_PORT=3307 등 환경변수
